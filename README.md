# Monitoring

Tampermonkey-скрипты для работы с платёжными тикетами в TH Management. Репозиторий устроен так, что скрипты **автообновляются** у всех, кто их установил — правишь код здесь, поднимаешь версию, пушишь в `main`, и у всех подтягивается новая версия сама.

## Установка

Нужно установить **оба** скрипта — они работают в паре (подробнее в разделе [«Как устроена система»](#как-устроена-система)).

Открыть ссылку в браузере с установленным [Tampermonkey](https://www.tampermonkey.net/) и подтвердить установку:

1. [TH Management — Bulk Approve Tickets (225)](https://raw.githubusercontent.com/partenkoEgor/Monitoring/main/scripts/th-bulk-approve.user.js) — массовая обработка тикетов (два режима: «225» и «239»)
2. [Auto-close SweetAlert2 success popups](https://raw.githubusercontent.com/partenkoEgor/Monitoring/main/scripts/th-autoclose-swal.user.js) — автозакрытие штатных попапов

После установки проверить в панели Tampermonkey, что оба скрипта **включены** (переключатель в списке).

---

## Как устроена система

Скрипты специально разделены на два независимых файла:

| Скрипт | Назначение | Где действует |
|---|---|---|
| **Bulk Approve Tickets (225 / 239)** | Массовая обработка тикетов, две кнопки на экране (225 и 239) | Только страницы `paymentsupport*` (3 домена) |
| **Auto-Close SweetAlert2** | Автоматически закрывает штатные попапы с «OK!» | Весь `backoffice/*` (3 домена) |

У Auto-Close более широкий охват — он нужен во всей админке, а не только во время массовых прогонов. При этом Bulk Approve имеет свой собственный резервный обработчик тех же попапов на случай, если Auto-Close почему-то не сработает вовремя. Это дублирование безвредно и сделано намеренно.

**Объединять их в один скрипт не стоит**: либо пришлось бы расширять `@match` кнопок Bulk Approve на весь бэкофис (кнопки вылезали бы везде), либо сужать Auto-Close только до paymentsupport (терялось бы автозакрытие попапов на остальных страницах).

## Bulk Approve Tickets (225 / 239)

`scripts/th-bulk-approve.user.js` — текущая версия: **2.2**.

### Что делает

Проходит по всем тикетам, видимым в таблице на текущей странице списка платёжных запросов, и для каждого:

1. Проверяет колонку **External Status** — если она не равна требуемому значению для выбранного режима, тикет пропускается.
2. Для режима **239** дополнительно проверяет колонку **Amount** — если сумма равна `0`, тикет тоже пропускается (см. ниже).
3. Для подходящих тикетов открывает Edit, ставит целевой статус в выпадающем списке Status и жмёт **Apply**.
4. Ждёт, пока модалка закроется (значит, сохранение прошло).
5. Ловит и логирует любые всплывающие окна (кроме штатного «OK!»), которые появляются в процессе — например, предупреждение о дублирующейся транзакции.
6. В конце показывает итоговую сводку и список тикетов, которые нужно проверить вручную.

На экране — две независимые кнопки запуска (режима), одна общая кнопка **СТОП** (одновременно может выполняться только один прогон):

| Кнопка | External Status тикета (фильтр) | Целевой статус | Доп. проверка |
|---|---|---|---|
| **Bulk Approve (225)** | `Approved (M)` | `225 Approved by agent` | — |
| **Bulk Response (239)** | `The money has not been sent, cancel it (M)` | `239 Response to user (M)` | Пропускает тикеты с `Amount = 0` |

Вся остальная механика (поиск полей, отлов попапов, отчёт, кнопка СТОП, поиск колонок по названию) — общая для обоих режимов.

### Пропуск тикетов с Amount = 0 (только режим 239)

У части тикетов 239 в колонке **Amount** стоит `0` — их не нужно обрабатывать автоматически. Для таких тикетов:

- Колонка **Amount** ищется по названию в шапке таблицы (как и остальные колонки), но резервного номера для неё нет: если колонка не нашлась, скрипт один раз предупредит об этом в консоли и обработает такие тикеты как обычно (без проверки на 0), а не рискнёт прочитать не ту ячейку.
- Значение парсится из текста ячейки (валютные символы, пробелы, разделители тысяч убираются).
- Тикет с `Amount = 0` помечается как `skipped`/`zero-amount` и **не** трогается.
- В финальном alert такие тикеты выносятся в отдельный список «Пропущены из-за Amount = 0, требуют ручной проверки» — аналогично списку дубликатов.

### Как запускать

1. Открыть страницу со списком тикетов, настроить нужные фильтры (даты, статус и т.д.) как обычно.
2. Нажать нужную кнопку в правом нижнем углу — **«Bulk Approve (225)»** или **«Bulk Response (239)»**, в зависимости от того, какие тикеты нужно обработать.
3. Подтвердить диалог с числом найденных на экране тикетов.
4. Скрипт последовательно обработает все строки. В любой момент можно нажать красную кнопку **«СТОП»**, которая появляется рядом с кнопками запуска на время прогона — скрипт корректно завершит текущий шаг (не обрывает на середине клика) и покажет промежуточный отчёт.
5. По завершении (или после СТОП) появится итоговое всплывающее окно со сводкой.

> **Важно:** обрабатываются только тикеты, физически загруженные на странице сейчас. Если тикетов больше, чем помещается на одной странице/фильтре — пагинация не обходится автоматически, нужно прогонять скрипт для каждой порции отдельно.

### Итоговый отчёт

```
Готово.
Успешно: 98
Пропущено (не тот External Status): 1
Пропущено (Amount = 0): 2
Ошибок: 1

Требуют ручной проверки на дубликаты (1):
19406922 - 19993860563

Пропущены из-за Amount = 0, требуют ручной проверки (2):
19406930
19406941

Подробности — в консоли (F12).
```

- **Успешно** — тикет обработан, модалка закрылась после Apply.
- **Пропущено (не тот External Status)** — External Status тикета не совпал с требуемым для этого режима, тикет не трогали.
- **Пропущено (Amount = 0)** — только для режима 239: у тикета Amount = 0, тикет не трогали. Строка показывается, только если такие тикеты были в этом прогоне.
- **Ошибок** — что-то пошло не так на одном из шагов (см. таблицу причин ниже, диагностика — в консоли).
- **Требуют ручной проверки на дубликаты** — тикеты, где всплыло нестандартное окно во время обработки (формат `Ticket ID - Transaction ID`). Их нужно открыть руками — вероятно, транзакция уже используется в другом обращении.
- **Пропущены из-за Amount = 0** — список Ticket ID для ручной проверки (только режим 239).

Если по конкретному блоку нечего показать (нет дублей или нет пропусков по сумме 0) — соответствующая часть в alert не показывается.

### Как ловятся попапы

Отдельный наблюдатель следит за всеми окнами SweetAlert2 (`.swal2-popup`):

- Штатное **«OK!»** (зелёная галочка, кнопка «OK», без кнопки отмены) — закрывается молча, без записи в отчёт.
- Любое **другое** окно — записывается в отчёт с привязкой к тикету и его Transaction ID.
  - Без кнопки отмены и **не** с иконкой ошибки — единственная доступная кнопка нажимается автоматически, чтобы не блокировать прогон, но текст всё равно попадает в отчёт.
  - С иконкой **error** (например, «Amount on receipt field must be filled in») — скрипт ничего не нажимает, оставляет окно на экране, чтобы можно было прочитать текст. Тикет в этом случае, скорее всего, попадёт в «Ошибки» (таймаут на закрытие модалки), а незакрытый попап нужно будет закрыть вручную перед тем, как скрипт перейдёт к следующему тикету.
  - С кнопкой отмены (сайт предлагает реальный выбор) — скрипт тоже ничего не нажимает, оставляет окно как есть.

Проверить детали после прогона можно в консоли (F12 → Console):

```js
// Таблица со всеми пойманными окнами (тикет, TXN ID, иконка, заголовок, текст, была ли кнопка отмены)
window.__bulkApproveCapturedPopups

// Полный список результатов по каждому тикету
window.__bulkApproveLastResults

// Найти конкретный тикет в результатах
window.__bulkApproveLastResults.find(r => r.ticketId === '19406922')
```

### Настройки (CONFIG и WORKFLOWS)

В начале файла скрипта есть блок `CONFIG` с общими для всех режимов таймингами:

```js
const CONFIG = {
  stepDelay: 250,             // пауза между шагами внутри одного тикета (мс)
  waitTimeout: 8000,          // сколько ждать появления/исчезновения элемента (мс)
  betweenTicketsDelay: 2000,  // пауза между тикетами (мс)
};
```

- Если Vue не успевает среагировать (элементы не находятся, хотя видны на экране) — увеличить `stepDelay` и/или `waitTimeout`.
- `betweenTicketsDelay` влияет только на скорость прогона; пропущенные тикеты эту паузу не ждут.

Ниже — массив `WORKFLOWS`, по одному объекту на каждую кнопку/режим:

```js
const WORKFLOWS = [
  {
    id: '225',
    buttonLabel: 'Bulk Approve (225)',
    buttonColor: '#2ABFCF',
    requiredExternalStatus: 'Approved (M)',  // какой External Status нужен, чтобы не пропустить тикет
    searchTerm: '225',                        // что печатается в поле поиска статуса
    statusMatch: (text) => { ... },           // как определяется нужная опция статуса
    targetStatusLabel: '225 Approved by agent', // только для текста диалогов/отчёта
  },
  {
    id: '239',
    // ... requiredExternalStatus / searchTerm / statusMatch / targetStatusLabel — та же схема
    skipZeroAmount: true, // пропускать тикеты, у которых Amount = 0 (см. раздел выше)
  },
];
```

Чтобы добавить ещё один режим (ещё одну кнопку) — достаточно добавить новый объект в `WORKFLOWS`: скрипт сам создаст для него кнопку на экране и подключит общую логику (поиск полей, отлов попапов, отчёт, кнопка СТОП, поиск колонок по названию).

### Диагностика ошибок по тикету

Каждая запись в `window.__bulkApproveLastResults` содержит `status` и, если это ошибка — `reason`:

| `reason` | Что произошло | Что проверить |
|---|---|---|
| `no-edit-link` | Не найдена кнопка/ссылка Edit в строке | Не поменялась ли колонка Actions или её содержимое |
| `modal-not-shown` | После клика Edit модалка не появилась за `waitTimeout` | Не изменилась ли верстка модалки; сайт может тормозить |
| `no-status-field` | В модалке не найдено поле Status | Изменился текст лейбла или структура `.form-add .input-group` |
| `no-multiselect-tags` | Не найден `.multiselect__tags` внутри поля Status | vue-multiselect мог измениться/обновиться |
| `dropdown-not-opened-or-no-match` | Список не раскрылся, или в нём нет опции целевого статуса этого режима | Проверить `statusMatch` соответствующего объекта в `WORKFLOWS`; в консоли рядом выводится HTML поля Status для отладки |
| `no-apply-button` | Не найдена кнопка Apply в модалке | Изменилась верстка блока кнопок `.filter.btn-block` |
| `modal-not-closed` | После клика Apply модалка не закрылась за `waitTimeout` | Скорее всего, всплыло окно с ошибкой/предупреждением — проверить `window.__bulkApproveCapturedPopups` для этого тикета |
| `exception` | Необработанное исключение во время обработки тикета | Смотреть полный текст ошибки в консоли (`console.error`) |

Любая ошибка не прерывает весь прогон целиком — скрипт помечает тикет как `failed` и переходит к следующему.

Записи со `status: 'skipped'` тоже содержат `reason` — `wrong-external-status` (не подошёл External Status) или `zero-amount` (только режим 239, Amount = 0).

**Важно:** для всех причин из таблицы выше, кроме `no-edit-link` и `modal-not-shown` (там модалка ещё не открывалась), скрипт после ошибки сам пытается закрыть модалку Edit кнопкой Cancel и ждёт (до 3 секунд) её фактического закрытия, прежде чем перейти к следующему тикету. Это защита от бага, когда незакрытая модалка «утекала» в обработку следующей строки — сайт при клике Edit на новом тикете не открывал новую модалку, а переиспользовал старую (всё ещё привязанную к предыдущему тикету), и скрипт применял статус/Apply не к тому тикету. Если закрыть модалку не удалось (например, поверх висит незакрытый попап с ошибкой) — об этом будет отдельное предупреждение в консоли, и в этом случае прогон стоит проверить вручную.

### Поиск колонок по названию

Скрипт не привязан жёстко к номерам колонок таблицы (`nth-child`). Перед каждым прогоном он:

1. Читает `<th>` в шапке таблицы (`.table-wrapper thead th`).
2. Строит карту «название колонки → её текущий порядковый номер».
3. Использует эту карту для поиска **Ticket ID**, **External status**, **Transaction ID** и **Actions** независимо от порядка колонок у конкретного пользователя.

Если название колонки не находится (шапка не прогрузилась, колонку скрыли/переименовали) — скрипт откатывается на резервный номер (сейчас: External status = 8, Ticket ID = 9, Transaction ID = 11, Actions = 3) и пишет предупреждение в консоль:

```
[BulkApprove] Не найдены колонки по названию: transaction id — для них будут использованы резервные номера.
```

Если видишь такое предупреждение — стоит проверить, не съехали ли резервные номера у конкретного пользователя (можно поправить их прямо в коде функции `getColumnIndex(...)`).

### Ограничения и важные предположения

- Таблица тикетов и модалка Edit рендерятся в основном DOM (не в iframe).
- Поле Status — компонент **vue-multiselect**; открытие списка завязано на фокус внутреннего `<input>`, а не на клик по обёртке.
- Сайт использует **SweetAlert2** для всплывающих окон (не нативные `alert()`/`confirm()`), кроме диалога подтверждения самого скрипта — он использует нативный `confirm()`.
- Обрабатываются только тикеты, уже загруженные на странице; пагинация не листается автоматически.
- Одновременно с этим скриптом может работать другой Tampermonkey-скрипт (авто-датапикер фильтров), который иногда делает полную перезагрузку страницы формой (`method="post"`) — это ожидаемо и не является ошибкой.

## Auto-close SweetAlert2 success popups

`scripts/th-autoclose-swal.user.js` — текущая версия: **1.7**.

### Что делает

Работает на всех страницах `backoffice/*` (не только paymentsupport) и следит за появлением попапов SweetAlert2. Закрывает автоматически **только** те, что строго соответствуют штатному подтверждению сохранения:

- иконка успеха (`swal2-icon-success`);
- содержимое — текст `"OK!"` (с учётом визуально похожих кириллических букв О/о/К/к, которые иногда попадают в текст по ошибке раскладки/копипаста);
- кнопка подтверждения с текстом `"OK"`;
- **нет** кнопки отмены.

Если хотя бы одно условие не выполняется — попап не трогается, чтобы случайно не закрыть окно с реальным предупреждением или ошибкой.

Работает независимо от Bulk Approve и нужен для всей админки: например, при обычном ручном сохранении карточек вне массовых прогонов.

### Диагностика

Каждое автозакрытие пишется в консоль (F12 → Console):

```
[AutoClose] Закрыл попап: title "...", содержимое "OK!", кнопка "OK"
```

С версии 1.7 в лог добавлен `title` — если когда-нибудь окно с реальным сообщением (например, валидационной ошибкой) начнёт закрываться автоматически, это будет видно по тексту title в этой строке, а не потеряется молча.

## Обновление скриптов

1. Внести изменения в нужный файл в `scripts/`.
2. Увеличить `@version` в шапке скрипта — без этого Tampermonkey не увидит обновление.
3. Запушить изменения в `main`.

Tampermonkey периодически (или по кнопке «Check for updates» в панели) сверяет `@version` по ссылке из `@updateURL` и подтягивает новый код с `@downloadURL`. Пока изменения не в `main` — установленные у пользователей копии их не увидят.
